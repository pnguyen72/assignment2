Module 3: Requirements Determination

A requirement is something the system should do, or a characteristic it should have.

Types of requirements

  • Business: reasons for proposing project
  • User: what the users need to do
  • Functional: what the system should do / what features it should have
  • Non-functional: characteristics the system should have
  • System: how the system should be built

These are not mutually exclusive. A feature/characteristic may fill multiple requirement types.

Subcategories of non-functional requirements

  1. Operational: physical and technical compatibility
    Examples:
    • run on the Steam Deck
    • integrable with the comany's existing system
  2. Performance: speed, capacity, availability
    Examples:
    • available 24/7
    • supporting 300 simultaneous users
  3. Security: who has authorized access under what circumstances
    Examples:
    • antivirus measures
    • only direct managers can see staff personnel records
  4. Cultural and Political: cultural, political, legal factors that affect the system
    Examples:
    • company policy is to only buy computers from Dell
    • supporting local currency and measurement units (*)

(*) I think this is an example of a feature that fits multiple requirement types because it also sounds like an FR.

How to discover requirements?

Common techniques

  • Interviews
  • Joint Application Development
  • Questionnaires
  • Document Analysis
  • Observation

Each technique has strengths and weaknesses. Most projects benefit from a combination of techniques.

Technique characteristics

  • Type of information: Understanding the as-is system, identifying improvements, or developing the to-be system.
  • Depth of information: How rich and detailed is the collected information. How useful is the technique at obtaining not only facts and opinions, but also why those facts and opinions exist.
    E.g., “What kind of currency option should be implemented in the payment system? (Depth on the topic of payment.)
  • Breadth of information: The range of information that can be collected.
    E.g., “When implementing a new system, what are your priorities for the payment system, inventory management, user interface, and customer support?" (Breadth, since this covers multiple topics.)
  • Integration of information: How easy it is to resolve conflicting info from different sources.
  • User involvement: The amount of time and energy the users must devote to the analysis process.
  • Cost: Self-explanatory.
